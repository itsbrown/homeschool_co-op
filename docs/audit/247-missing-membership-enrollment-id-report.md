## Section 5 — Production migration plan

The schema change is purely additive — a nullable `integer` column with a new FK and no default — so it is online-safe for Postgres without rewriting the table.

### 5a. Pre-deploy verification

1. Confirm production `\d payment_allocations` does **not** already have `membership_enrollment_id` (the column may have been added manually).
2. Capture production `SELECT COUNT(*) FROM payment_allocations;` for the audit-trail backlog (input to a future heuristic-correlation task, see §2).

### 5b. Migration DDL (idempotent)

```sql
ALTER TABLE payment_allocations
  ADD COLUMN IF NOT EXISTS membership_enrollment_id integer;

DO $$
BEGIN
  IF NOT EXISTS (
    SELECT 1 FROM pg_constraint
    WHERE conname = 'payment_allocations_membership_enrollment_id_fkey'
  ) THEN
    ALTER TABLE payment_allocations
      ADD CONSTRAINT payment_allocations_membership_enrollment_id_fkey
      FOREIGN KEY (membership_enrollment_id)
      REFERENCES membership_enrollments(id) ON DELETE CASCADE;
  END IF;
END $$;
```

Both `ADD COLUMN` (nullable, no default) and `ADD CONSTRAINT FOREIGN KEY` are online operations in Postgres — they take a brief `ACCESS EXCLUSIVE` lock to update the catalog but do not rewrite the table. The FK constraint is validated against existing rows; with the column NULL on every existing row, validation is effectively a no-op.

### 5c. Post-deploy verification

```sql
\d payment_allocations
-- expect: membership_enrollment_id column + FK to membership_enrollments(id) ON DELETE CASCADE

-- Smoke check the next reallocation:
SELECT id, payment_history_id, enrollment_id, membership_enrollment_id, allocation_type, created_at
FROM payment_allocations
ORDER BY id DESC LIMIT 5;
```

### 5d. Rollback

```sql
ALTER TABLE payment_allocations
  DROP CONSTRAINT IF EXISTS payment_allocations_membership_enrollment_id_fkey;
ALTER TABLE payment_allocations
  DROP COLUMN IF EXISTS membership_enrollment_id;
```

Safe so long as no application writes have populated the column. Production reallocation traffic should be paused (or the deploy reverted) before rollback.

### 5e. Out of scope for this migration

- Backfilling historical `payment_allocations` rows with the correct `membership_enrollment_id`. To be filed as a separate task once a best-effort heuristic correlation has been designed (per §2).
- Fixing the legacy `enrollment_id` FK target (`school_class_enrollments` → `program_enrollments`). Tracked separately as Task #246.
- Upgrading drizzle-kit to one that can parse `COALESCE(...)` partial-expression indexes so `npm run db:push` succeeds end-to-end on this dev DB.
