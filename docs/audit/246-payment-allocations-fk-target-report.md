## Remediation chosen — Option A

`payment_allocations.enrollment_id` is dropped + re-added to reference
`program_enrollments(id)`. Rationale:

1. **Every write site already writes `program_enrollments(id)` values.** Audited
   below: `PaymentReallocationService` (out/in audit pair), `PaymentProcessorService`
   (positive allocations), `webhook-handler.ts` refund rollback (3 negative
   allocation paths). All five inserts feed the column an id that originates
   from `program_enrollments` (either `enrollment.id` from `getProgramEnrollmentById`,
   or `program_enrollments` rows looked up by `getAllEnrollments`/`storage.updateProgramEnrollment`).
2. **No write site references `school_class_enrollments(id)`** for this column.
   The original FK target was therefore a copy/paste hazard, not an indication
   of a dual-table reference.
3. **Option B (split column + CHECK)** would be over-engineering for a column
   that is only ever fed program_enrollments ids — and the sibling
   `membership_enrollment_id` column already covers the membership-only path.

```sql
  WHERE NOT EXISTS (SELECT 1 FROM program_enrollments    pe  WHERE pe.id  = pa.enrollment_id)
    AND NOT EXISTS (SELECT 1 FROM school_class_enrollments sce WHERE sce.id = pa.enrollment_id);
```

Decision matrix:

- `in_school_class_only > 0` → **STOP**. Some live rows actually point at
  `school_class_enrollments`. Repointing the FK at `program_enrollments`
  would reject those rows. Escalate before applying — this contradicts the
  audit done in dev and means a write site this report missed exists.
- `in_both > 0` (id collisions) → proceed with caution and snapshot the
  affected rows for post-migration spot-check; the join semantics will
  change from "school class enrollment X" to "program enrollment X" for
  those rows.
- `in_program_only` and `orphaned` → safe to migrate. (Orphaned rows
  predate the dropped-FK incident referenced in #222 and will be rejected
  by the new FK. Either delete them under a separate cleanup task or
  reattach them to the right `program_enrollments` row first.)

### Migration SQL (production)

```sql
BEGIN;
-- Lock acquired is brief — Postgres can drop+add a FK without rewriting the
-- table. The ADD CONSTRAINT validates every existing row; ensure the
-- pre-flight returned zero rows under in_school_class_only and zero
-- orphaned rows before running this.
ALTER TABLE payment_allocations
  DROP CONSTRAINT payment_allocations_enrollment_id_fkey;
ALTER TABLE payment_allocations
  ADD  CONSTRAINT payment_allocations_enrollment_id_fkey
       FOREIGN KEY (enrollment_id)
       REFERENCES program_enrollments(id) ON DELETE CASCADE;
COMMIT;
```

For very large `payment_allocations` tables, the validation step can be
made online with `NOT VALID` + `VALIDATE CONSTRAINT`:

```sql
ALTER TABLE payment_allocations
  ADD CONSTRAINT payment_allocations_enrollment_id_fkey
      FOREIGN KEY (enrollment_id)
      REFERENCES program_enrollments(id) ON DELETE CASCADE
      NOT VALID;
ALTER TABLE payment_allocations
  VALIDATE CONSTRAINT payment_allocations_enrollment_id_fkey;
```

`VALIDATE CONSTRAINT` only takes a `SHARE UPDATE EXCLUSIVE` lock and
scans the table in the background — safe under production write load.
