column name, but for the COALESCE expression column it gets `null`, hence the
ZodError.

### 1.3 Option chosen and rationale

**Option A** (replace the expression index with two partial UNIQUE indexes).

- **Why A:** The semantic guarantee of the legacy index is "for any (user_id,
  role) pair, allow at most one row per `school_id` value, treating NULL as a
  single distinct value." That decomposes cleanly into:
  - `UNIQUE (user_id, role, school_id) WHERE school_id IS NOT NULL` — uniqueness
    when a school is named.
  - `UNIQUE (user_id, role)              WHERE school_id IS NULL`     — at most
    one school-less row per (user_id, role).
  Both use plain column references that drizzle-kit can introspect, and Drizzle
  ORM exposes `uniqueIndex(...).on(...).where(sql\`...\`)` so the constraint
  lives in `shared/schema.ts` (no schema/database drift).
- **Why not B (upgrade drizzle-kit):** drizzle-kit 0.30.x is the version pinned
  in this repo (`package.json` line 181). A blind upgrade to 0.31+ could
  introduce other introspection regressions across this 3617-line schema and
  is a larger surface than necessary. The bug is upstream and tracked by
  drizzle (issues filed under "expression index introspection"), but switching
  major versions is out of scope for a targeted hotfix.
- **Why not C (manual-SQL policy + CI gate):** Three tasks in a row already
  resorted to hand-rolled SQL because db:push was broken; codifying that as
  policy entrenches the brittleness instead of fixing the root cause. Option A
  also turns out to require a small additional fix (export the `roleEnum`,
  Section 2.5) which a CI gate would not have caught.

## Section 2 — Apply the fix

### 2.1 New file: `server/migrations/fix-user-roles-unique-index.sql`

```sql
-- Migration: Fix user_roles unique index for drizzle-kit compatibility
-- Date: 2026-05-12
-- Task: #242
--
-- Replaces idx_user_roles_unique_user_role (UNIQUE on user_id, role, COALESCE(school_id, 0))
-- with two partial UNIQUE indexes that drizzle-kit can introspect (no expression columns).
--
-- Original semantic: a user cannot hold the same role twice in the same school context,
-- and NULL school_id is treated as duplicate of NULL school_id (because COALESCE(NULL, 0)
-- collapses NULLs to 0). The two partial indexes preserve this exactly:
--   * (user_id, role, school_id) WHERE school_id IS NOT NULL  -- per-school uniqueness
--   * (user_id, role)              WHERE school_id IS NULL    -- one global-NULL row per user/role
--
-- Edge case difference from the legacy index: a row with school_id = 0 and a row with
-- school_id = NULL were treated as duplicates by the COALESCE expression. Under the new
-- partial indexes they are NOT duplicates. school_id=0 is not a valid school PK in this
-- schema (serial starts at 1), so this case does not occur in practice.

BEGIN;

DROP INDEX IF EXISTS idx_user_roles_unique_user_role;

CREATE UNIQUE INDEX IF NOT EXISTS idx_user_roles_unique_user_role_school
  ON user_roles (user_id, role, school_id)
  WHERE school_id IS NOT NULL;

CREATE UNIQUE INDEX IF NOT EXISTS idx_user_roles_unique_user_role_no_school
  ON user_roles (user_id, role)
  WHERE school_id IS NULL;

COMMIT;
```

### 4.2 Duplicate (user_id, role) WHERE school_id IS NULL — must be rejected

```
$ psql -c "INSERT INTO user_roles (user_id, role, school_id) VALUES (9001, 'parent', NULL);"
INSERT 0 1
$ psql -c "INSERT INTO user_roles (user_id, role, school_id) VALUES (9001, 'parent', NULL);"

```

✅ NULL/NULL collision preserved (default Postgres NULL=NULL semantics would
let two NULLs coexist; the partial index forbids it because the partial
predicate matches both rows and the indexed columns are equal).

### 4.3 NULL vs school_id = 0 — semantic *change* (documented)

```
$ psql -c "INSERT INTO user_roles (user_id, role, school_id) VALUES (9001, 'admin', 0);"
INSERT 0 1
$ psql -c "INSERT INTO user_roles (user_id, role, school_id) VALUES (9001, 'admin', NULL);"
INSERT 0 1
$ psql -c "SELECT user_id, role, school_id FROM user_roles WHERE user_id=9001 AND role='admin' ORDER BY school_id NULLS LAST;"
 user_id | role  | school_id
---------+-------+-----------
    9001 | admin |         0
    9001 | admin |
(2 rows)
```

Under the legacy `COALESCE(school_id, 0)` index, the second insert would have
been rejected because `COALESCE(NULL, 0) = COALESCE(0, 0) = 0`. Under the
partial-index pair, `school_id = 0` and `school_id IS NULL` are now distinct.

**This is an intentional, documented semantic change. It is safe in practice
because:**

- `schools.id` is `serial PRIMARY KEY`; Postgres serials start at 1, so a
  `school_id = 0` row in `user_roles` would dangle (no valid school FK target)
  and is not produced by any code path.
- A check (`SELECT … WHERE school_id = 0`) on the dev DB returned 0 rows
  (Section 1's pre-flight check on the live data also confirmed no
  `(user_id, role, school_id)` duplicates exist).
- Any future row with `school_id = 0` would still be FK-validated against
  `schools` (which has no id 0), so the schema continues to reject the
  scenario at the FK level even without the historical COALESCE collapse.

A follow-up task (proposed) covers a one-time defensive scan + tightening
should this assumption change.
