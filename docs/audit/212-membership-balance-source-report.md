## Section 0: What changed

`server/utils/cart-pricing.ts`
- Two new exported pure helpers (so the integration test and the helper test can both exercise the same code path):
  - `findUnpaidMembershipRow(memberships, schoolId, currentYear)` — picks the unpaid `membership_enrollments` row for `schoolId` matching the current or next membership year. Skips `enrolled` / `grace_period` (already paid) and `expired` / `suspended` (never re-charge).
  - `computeUnpaidMembershipRemainingCents(row)` — returns the row's remaining balance in cents. Prefers `remainingBalance`, falls back to `amount - amountPaid`. Anomalies (overpayment, NULL paid info, NaN) clamp to **0** and emit a `WARN` log — they MUST NOT silently fall back to the school's full fee (the bug task #212 fixes).
- `calculateCartSnapshot` now calls those helpers. When an unpaid membership row exists, the membership line uses the row's remaining balance — NOT `school.membershipFeeAmount`.
- First-time enrollment path (no row) is unchanged and still applies the optional membership discount.

Tests (both passing — see Section 6):
- `server/tests/cart-snapshot-membership-balance.test.ts` — 20 unit tests against the helpers.
- `server/tests/cart-snapshot-membership-balance.integration.test.ts` — 6 **integration** tests that seed real `membership_enrollments` rows in the dev DB, call `calculateCartSnapshot` end-to-end, capture each raw snapshot to `docs/audit/212-evidence/`, and assert on `snapshot.totals.membershipTotal` / `grandTotal`.

## Section 7: Test architecture notes

The integration test inserts a synthetic parent (`users.id = 999912`, role `parent`) via direct Drizzle insert in `beforeAll` so `membership_enrollments.parentUserId` FK is satisfied; school 1's `membershipFeeAmount` is force-set to `17500` and restored in `afterAll`; per-test `cleanupTestMemberships` removes any rows created for the synthetic parent so each property runs in isolation. All seeded rows are deleted and the synthetic user is removed at the end of the suite, so the test leaves no residue in the dev DB.

Anomaly cases that the DB schema cannot represent (NULL `amountPaid`/`remainingBalance` — both columns are `NOT NULL`) are exercised at the helper level by the unit suite. The DB-representable anomaly (negative `remainingBalance`) is exercised end-to-end by the integration suite.

## Section 8: Disputed-status edge case (added per code review feedback)

**Property 6 — `grace_period` row WITH positive `remainingBalance` → snapshot membershipTotal = 0**

The reviewer flagged a coverage gap around statuses outside `pending_payment`. The product semantic for `grace_period` is locked in `shared/schema.ts`:

```
VALID_PAID_MEMBERSHIP_STATUSES = ['enrolled', 'grace_period']
// "grace_period: Membership expired but still within grace period (counts as active)"
```

`isActiveMembership('grace_period')` returns `true` → `alreadyPaid` short-circuit → `membershipTotal = 0`. We must NOT use the row's `remainingBalance` here, otherwise a parent inside their renewal window would be double-charged via the cart while their renewal is being handled separately.

Property 6 (added test) seeds a `grace_period` row with `remainingBalance = 12500` and asserts:
- `snapshot.totals.membershipTotal === 0` (NOT `12500`, NOT the full fee `17500`)
- `snapshot.membership.alreadyPaid === true`

Raw evidence: `docs/audit/212-evidence/property6-grace-period-with-balance.json` and the live log line:
`🎫 Active membership found: true { year: 2026, status: 'grace_period', remainingBalance: 12500 }` (no `🎫 Cart snapshot using unpaid membership row remaining balance` log emitted, confirming the unpaid-row helper was not invoked for this status).

**Out-of-scope UI inconsistency:** `client/src/utils/parentBalance.ts:getMembershipOutstandingBalance` only skips `expired`/`suspended`, so it would currently return `12500` for the same row. That is a pre-existing UI helper inconsistency NOT introduced by this task and is captured as follow-up #243 — the cart server is the authority for what is actually charged, and it correctly returns 0 here.
