## Section 2 — Heuristic definition (exact SQL)

The full backfill SQL is committed at
`server/migrations/248-backfill-payment-allocations-membership-enrollment-id.sql`
(idempotent, dry-run by default — see §3). The heuristic at its core is:

```sql
WITH candidate_pairs AS (
  SELECT
    pa.id            AS allocation_id,
    pa.created_at    AS allocation_created_at,
    pa.allocated_amount_cents,
    sph.user_id      AS parent_user_id,
    me.id            AS membership_enrollment_id
  FROM payment_allocations pa
  JOIN stripe_payment_history sph ON sph.id = pa.payment_history_id
  JOIN membership_enrollments me
    ON me.parent_user_id = sph.user_id
   AND me.created_at <= pa.created_at + interval '7 days'
   AND me.amount IS NOT NULL
   AND me.amount = pa.allocated_amount_cents
  WHERE pa.membership_enrollment_id IS NULL
    AND pa.allocation_type = 'membership'
    AND pa.enrollment_id   IS NULL
),
counts AS (
  SELECT allocation_id,
         COUNT(*)                     AS n_candidates,
         MIN(membership_enrollment_id) AS only_candidate_id
  FROM candidate_pairs
  GROUP BY allocation_id
)
-- UNIQUE-MATCH bucket: n_candidates = 1
-- AMBIGUOUS_SKIP bucket: n_candidates > 1
-- UNRECOVERABLE_NULL bucket: eligible_null_rows - unique - ambiguous
SELECT * FROM counts;
```

### Filters explained

| Filter | Why this clause is required |
| ------ | --------------------------- |
| `pa.membership_enrollment_id IS NULL` | Don't overwrite already-populated post-#247 rows. Makes the script idempotent. |
| `pa.allocation_type = 'membership'`   | Only the `'membership'` allocation_type is unambiguously a membership-paying row. The `'payment'` and `'reallocation_*'` rows are class-enrollment side and MUST stay NULL. Audited write sites: `server/api/scheduled-payments.ts:1640-1645` and `:2608-2613` (the only two sites in the repo that ever set `membershipEnrollmentId` to a non-null value when creating an allocation). |
| `pa.enrollment_id IS NULL`            | The two membership-write sites above always set `enrollmentId: null` alongside the membership id. Restricting to NULL `enrollment_id` excludes any row where a class-enrollment payment was mis-typed as 'membership' — preserving the "one of enrollment_id or membership_enrollment_id is set" invariant from `shared/schema.ts:1142-1144`. |
| `me.parent_user_id = sph.user_id`     | Walks the only persistent linkage: `payment_allocations.payment_history_id → stripe_payment_history.user_id` is the parent who paid, and a membership belongs to a parent. (`stripe_payment_history` has a UNIQUE constraint on `payment_intent_id` so the `payment_history_id → user_id` mapping is well-defined.) |
| `me.created_at <= pa.created_at + interval '7 days'` | The membership row must have existed at or near allocation time. Forward 7-day tolerance covers the documented webhook race where the allocation insert can race ahead of the membership update under load. NO backward tolerance — an allocation cannot retroactively pay a membership created weeks later (those would be a different membership year). |
| `me.amount IS NOT NULL AND me.amount = pa.allocated_amount_cents` | EXACT amount match — no fuzzy money tolerance. A membership of $200 paid via biweekly installments would create a `payment_allocations` row of $200 (the membership-priority disbursement allocates the full membership amount in one go — see §247 audit and `server/api/scheduled-payments.ts:1644 perPaymentMembershipAmount`). Partial-membership rows would not match exactly and fall into UNRECOVERABLE_NULL — a deliberately conservative choice that preserves accuracy at the cost of recall. |

### Per-clause false-positive / false-negative analysis

1. **`me.parent_user_id = sph.user_id`**
   - **False-positive scenario:** none on this column alone — parent ownership is a hard scope.
   - **False-negative scenario:** a payment made by a guardian whose `users.id` differs from the membership's `parent_user_id` (multi-guardian system). Acceptable: those rows fall into UNRECOVERABLE_NULL and are left for manual reconciliation (separate task, per §6).

2. **`me.created_at <= pa.created_at + interval '7 days'`**
   - **False-positive scenario:** a parent has TWO memberships (e.g. two consecutive school years) whose `created_at` both fall within 7 days of the same allocation. Eliminated by the AMBIGUOUS_SKIP bucket — those rows are NOT updated.
   - **False-negative scenario:** a webhook arrived >7 days after a membership row was updated (extremely unlikely; Stripe retries cap at ~3 days). Acceptable: UNRECOVERABLE_NULL.

3. **`me.amount = pa.allocated_amount_cents` (exact)**
   - **False-positive scenario:** two memberships for the same parent with the same exact amount and overlapping 7-day windows — eliminated by AMBIGUOUS_SKIP.
   - **False-negative scenario:** a school changed its membership fee mid-year and the `me.amount` recorded today differs from what was actually charged. Acceptable: UNRECOVERABLE_NULL — exact-amount is the only safe rule for money.

- `allocation_type IN ('payment', 'reallocation_in', 'reallocation_out', 'adjustment', 'refund')`

  upstream bug; this backfill leaves them alone and a follow-up task can

## Section 5 — Rollback plan + verification

### Rollback strategy

Stage 2 captures the prior `(id, prior_membership_enrollment_id)` of every
row it touches into a temp table named `_t248_snapshot` BEFORE the UPDATE
runs (see committed SQL, step 1). For a live prod run the operator promotes
this temp table to a real table BEFORE COMMIT — the simplest path is:

```sql
BEGIN;
-- Step A: re-run the backfill SQL up to the UPDATE, but on a real table:
CREATE TABLE backfill_248_snapshot (
  id integer PRIMARY KEY,
  prior_membership_enrollment_id integer,
  proposed_membership_enrollment_id integer NOT NULL,
  applied_at timestamp DEFAULT now()
);
INSERT INTO backfill_248_snapshot (id, prior_membership_enrollment_id, proposed_membership_enrollment_id)
SELECT pa.id, pa.membership_enrollment_id, candidates.only_candidate_id
FROM   payment_allocations pa
JOIN   ( /* heuristic CTE returning unique matches only */ ) candidates
  ON   candidates.allocation_id = pa.id
WHERE  pa.membership_enrollment_id IS NULL;

-- Step B: apply the UPDATE.
UPDATE payment_allocations pa
SET    membership_enrollment_id = s.proposed_membership_enrollment_id
FROM   backfill_248_snapshot s
WHERE  pa.id = s.id AND pa.membership_enrollment_id IS NULL;

COMMIT;
```

Rollback is then a single transaction:

```sql
BEGIN;
UPDATE payment_allocations pa
SET    membership_enrollment_id = s.prior_membership_enrollment_id
FROM   backfill_248_snapshot s
WHERE  pa.id = s.id;
-- Optional: DROP TABLE backfill_248_snapshot;
COMMIT;
```

This is safe because:

- It only touches `(id IN backfill_248_snapshot)` — the exact set the
  backfill modified — and restores the prior value column-for-column.
- It does NOT depend on knowing the proposed value, so even if the
  heuristic changes later the rollback is still a faithful restore.
- It's idempotent: re-running it sets the same NULLs back to NULL.

### Abort decision owner

The on-call payment-safety reviewer owns the abort decision at every
checkpoint:
- Dry-run numbers don't pass the conservation invariant
  (`unique + ambiguous + unrecoverable != eligible`) → abort.
- Canary spot-check shows a wrong membership linked → abort and rollback
  the canary via the §5 SQL.
- Stage 2 FK validity check returns >0 → the migration's `ON_ERROR_STOP`
  aborts the transaction automatically; no rollback needed beyond letting
  the BEGIN/ROLLBACK unwind.

### Rollback trigger conditions

- Any of the §6 abort conditions above.
- Post-deploy report from a parent or admin that a membership refund or
  reallocation references the wrong membership.
- A discovery in a follow-up audit that the heuristic produced false
  positives in production (e.g. a parent with two memberships of identical
  amounts within a 7-day window who was previously believed unique).

Rollback SQL is the single transaction pasted in §5.
