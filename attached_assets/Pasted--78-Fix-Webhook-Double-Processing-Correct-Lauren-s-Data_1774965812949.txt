#78 - Fix Webhook Double-Processing & Correct Lauren's Data (Emergency)

What & Why
Stripe fires both checkout.session.completed and payment_intent.succeeded for the same cart checkout. Both handlers currently process enrollment payments, and the deduplication guard is racy. This caused daggs.lauren@gmail.com's records to show $900 paid when only $600 was collected. The same root cause also breaks Payment History for multi-child checkouts (only first child's name and wrong amount saved).

Done looks like
- Lauren's data is corrected: Gabriel and Jack Hamilton each show $300 paid, $600 remaining; Jack's April 27 installment is pending.
- Payment History shows the full $600 checkout payment with both children listed correctly.
- checkout.session.completed is the single source of truth for cart-checkout payments.
- payment_intent.succeeded skips checkout-originated payments (identified by checkout_session_id or metadata.paymentType).
- All updates (enrollment, scheduled payments, payment record, allocations) happen in a single transaction.
- Idempotency is atomic using stripe_payment_history idempotency_key + unique constraint.
- New admin endpoint PATCH /api/admin/enrollments/:id/correct-balance for future corrections with full audit log.

Out of scope
- Full webhook rewrite or new PaymentProcessorService
- Refunding via Stripe
- Broader audit of other families
