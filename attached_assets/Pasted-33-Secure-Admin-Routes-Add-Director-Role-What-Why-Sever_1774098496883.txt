33 - Secure Admin Routes & Add Director Role
What & Why
Several admin API routes are mounted without authentication middleware, and an open /api/auth/update-role endpoint allows unauthenticated role changes. Additionally, a director role is needed that can access the schedule builder (read and write) like a schoolAdmin, but is scoped to educator-facing views rather than the full admin panel. This task closes the security gaps found in the permissions audit and adds the director role end-to-end.

Done looks like
POST /api/auth/update-role requires a valid Supabase session with schoolAdmin or above, enforces that a caller cannot grant a role higher than their own, and correctly updates both users.role and the user_roles table entry as the source of truth
/api/admin, /api/admin-users, /api/admin-classes, /api/school-admin, and /api/billing router mounts all have supabaseAuth + requireRole applied at the mount point in routes.ts
The duplicate unprotected GET /api/knowledge-bases/:id route is removed; only the authenticated version remains
"director" is added to the roleEnum pgEnum in shared/schema.ts
An idempotent ALTER TYPE role ADD VALUE IF NOT EXISTS 'director' statement is added to server/init-db.ts so the enum change is reflected in the live database on next startup
All requireRole arrays in schedule-builder.ts (every GET, POST, PATCH, DELETE) include 'director' — director is NOT added to the roleHierarchy object in auth0-auth.ts (access is explicit, not hierarchical)
The EducatorAppShell.tsx sidebar shows a "Schedule Builder" nav item when activeRole === 'director'
No existing functionality for any other role is broken
Out of scope
Fixing individual unprotected routes inside admin-users.ts beyond the mount-point gateway (those are a separate audit item)
Changing the roleHierarchy object in auth0-auth.ts for director (explicit per-route access only)
Adding a director-specific UI page or dedicated dashboard
Tasks
Fix /api/auth/update-role — Add supabaseAuth and requireRole(['superAdmin', 'admin', 'schoolAdmin']) middleware to the route. Inside the handler:

Read the calling user's role from req.user?.id (the database integer ID set by supabaseAuth) — do NOT trust the request body for the caller's role
Apply a hierarchy check: superAdmin can assign any role; admin can assign anything except superAdmin; schoolAdmin can only assign teacher, director, educator, parent, or student — return 403 with a clear message otherwise
On success, update both users.role (legacy field) AND the relevant user_roles table entry (source of truth) using req.auth.dbUserId for database queries, not the Supabase UUID
Add a comment explaining the dual-write pattern
Secure router mount points in routes.ts — Add supabaseAuth and requireRole(['superAdmin', 'admin', 'schoolAdmin']) before each of these routers: /api/admin, /api/admin-users, /api/admin-classes, /api/school-admin, /api/billing. Add a comment on each mount line explaining the security intent. Where a router already applies supabaseAuth internally on individual routes, the mount-point guard is additive and safe — do not remove internal guards.

Remove duplicate unprotected knowledge-base route — Delete the first app.get("/api/knowledge-bases/:id", ...) definition (no auth middleware, around line 1012 in routes.ts). Keep only the authenticated definition. Add a brief comment explaining the duplicate was removed to prevent the unauthenticated handler from shadowing the protected one.

Add director to schema and run migration — In shared/schema.ts, add "director" to the roleEnum pgEnum array, placed between "schoolAdmin" and "teacher". Then add an idempotent migration statement to server/init-db.ts following the project's established pattern (since db:push is blocked by a drizzle-kit bug in this environment):

ALTER TYPE role ADD VALUE IF NOT EXISTS 'director';
Wrap it in a try-catch block consistent with the existing migration blocks in init-db.ts.

Add director to all requireRole arrays in schedule-builder.ts — For every route in the file (GET skeletons, GET skeletons/:id, POST skeletons, PATCH skeletons/:id, DELETE skeletons/:id, GET skeleton blocks, POST skeleton blocks, PATCH/DELETE blocks, GET week plans, POST week plans, PATCH week plans, and any other routes), add 'director' alongside 'schoolAdmin' in the allowed roles arrays. Do NOT add director to the roleHierarchy object in auth0-auth.ts — access must remain explicit per-route. Add a brief comment on each requireRole call group indicating director has schedule-builder-level access.

Show "Schedule Builder" nav item for director in EducatorAppShell.tsx — Use the existing useRole() hook (already imported) to read activeRole. Add a "Schedule Builder" nav item (linking to /educator/week-plans, the existing schedule route) that renders when activeRole === 'director'. Use an appropriate icon from lucide-react (e.g. LayoutGrid). Keep all existing nav items untouched.
