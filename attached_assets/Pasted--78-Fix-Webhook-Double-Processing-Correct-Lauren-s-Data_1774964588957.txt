#78 - Fix Webhook Double-Processing & Correct Lauren's Data (Emergency)

What & Why
Stripe fires both checkout.session.completed and payment_intent.succeeded for the same cart checkout. Both handlers currently process enrollment payments, and the deduplication guard is racy under concurrent delivery. This caused Lauren daggs.lauren@gmail.com's records to show $900 paid when only $600 was collected.

Done looks like
- Lauren's data is corrected: Gabriel and Jack Hamilton each show $300 paid, $600 remaining; Jack's April 27 installment is pending.
- checkout.session.completed is the single source of truth for cart-checkout payments.
- payment_intent.succeeded skips checkout-originated payments (identified by checkout_session_id or metadata.paymentType).
- Idempotency is atomic (uses stripe_payment_history + unique constraint or processed_webhook_events table).
- All enrollment + scheduled payment updates happen in a single transaction.
- New admin endpoint PATCH /api/admin/enrollments/:id/correct-balance for future corrections with audit log.

Out of scope
- Full webhook rewrite or new PaymentProcessorService
- Refunding via Stripe
- Broader audit of other families

Tasks

1. Correct Lauren's data (one-time script or admin endpoint run)
   - In a transaction:
     - Set Gabriel Hamilton enrollment: totalPaid = 30000, remainingBalance = 60000
     - Set Jack Hamilton enrollment: totalPaid = 30000, remainingBalance = 60000
     - Mark Jack's April 27 scheduled payment back to 'pending'
   - Log the correction with reason "Webhook double-processing fix for Lauren daggs.lauren@gmail.com"

2. Strengthen idempotency & ownership in webhook-handler.ts
   - In payment_intent.succeeded handler:
     - Early exit if payment has checkout_session_id or metadata.paymentType === 'cart_checkout'
     - Use stripe_payment_history idempotency key + unique constraint for deduplication
   - Wrap all enrollment + scheduled payment updates in db.transaction
   - Add log when duplicate is skipped: console.log(`Duplicate webhook skipped: ${event.id}`)

3. Add admin correction endpoint (server/api/admin-enrollments.ts or similar)
   - PATCH /api/admin/enrollments/:id/correct-balance
   - Body: { totalPaidCents: number, syncSchedule: boolean, reason: string }
   - Requires admin or director role
   - Updates enrollment, recomputes remainingBalance, optionally syncs scheduled payments
   - Logs correction to a new audit table or stripe_payment_history metadata

4. Add functional unique constraint (server/init-db.ts)
   - CREATE UNIQUE INDEX IF NOT EXISTS stripe_payment_history_idempotency_idx ON stripe_payment_history (idempotency_key) WHERE idempotency_key IS NOT NULL;

Architectural constraints
- All DB changes via IStorage (no raw queries in handlers)
- Migrations via idempotent SQL in init-db.ts only
- Use db.transaction for atomicity
- Log every correction with who, when, before/after, reason
