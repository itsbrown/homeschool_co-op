Key design decisions (short)

Role assignments are scoped (role + scope_type + scope_id). Example: role='location_coordinator', scope_type='location', scope_id=42.
Enforce with DB RLS so even if a bug in the app occurs, unauthorized users cannot read protected rows.
Provide server-side convenience endpoints that call the DB but rely on RLS for enforcement.
Audit all accesses of PII (who asked for what and when).
Admin UI + API to manage role assignments (assign/revoke).

Admin UI & APIs to manage role assignments
Provide endpoints:
POST /admin/roles/assign { userId, role, scope_type, scope_id }
POST /admin/roles/revoke { assignmentId } or DELETE
GET /admin/roles?userId=...
Ensure only admins can call these endpoints. Audit role assignment changes in access_logs (or dedicated role_change_logs).

Privacy & compliance considerations
Limit returned PII fields. Only include the minimal parent contact fields required.
Log access to PII (we added access_logs).
Consider short-lived signed URLs or masking phone/email in some UIs if full contact not needed.
Add retention rules for access_logs (e.g., purge older than X years if required).
Train coordinators about acceptable use of PII and add audit-review process.
Rollout strategy (safe)
Step 1: Add role_assignments and access_logs tables + admin UI to assign roles. Keep RLS off initially.
Step 2: Implement app-level checks in endpoints (like the example above) and enforce who can view data at the app level. Deploy and let coordinators use it.
Step 3: Add RLS policies carefully in staging. Test all flows and admin tasks. Ensure your admin user(s) still have access.
Step 4: Enable RLS in production and remove app-level bypasses (or keep them as UX checks). RLS should be final safety net.
Step 5: Audit logs review after rollout.
Practical notes for Supabase
If you host DB on Supabase, you can similarly add policies that use auth.uid() to match role_assignments.user_id.
You can also add computed JWT claims (custom claims) to include role in tokens to allow simpler policies (but scope-based claims can be large; better to keep role_assignments table and policies referencing it).
